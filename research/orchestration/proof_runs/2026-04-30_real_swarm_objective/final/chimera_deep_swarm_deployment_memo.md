# Chimera Deep Swarm Deployment Memo

Date: 2026-04-30

## Objective

Decide how Chimera should use the deep swarm in real work, how heartbeats and automations should behave around it, how the same logic should adapt across Codex, Claude/Cowork, OpenClaw/Kimi, and Hermes, and how a human should see the orchestration clearly.

## What Actually Ran

This was a real bounded swarm-backed internal objective, not a toy classification test.

What ran:

1. a written `plan.md`
2. a written `landscape_brief.md`
3. four dimension workers using the enforced verdict schema
4. a stronger cross-verifier
5. targeted validation on artifact completeness and policy distinctions
6. insight extraction
7. final writing and visual workflow assembly

What did not run:

- no recurring heartbeat for this objective
- no always-on automation for this objective

Why:

- this run was bounded and could be finished in one session
- heartbeats are for continuation, not for proving that the orchestration is real

## Recommendation

### Use Deep Swarm For

- weekly BTC or broader market theses
- major event-driven setup reviews
- deep ambiguity that needs cross-verification
- post-trade failure analysis
- large token, sector, or framework comparisons
- any research-heavy memo where contradictions matter

### Do Not Use Deep Swarm For

- routine setup watcher cycles
- position management loops
- every five-minute or fifteen-minute market wake
- one-step monitor or repair tasks
- normal coding objectives

## Platform Split

### Windows Codex

Use for:

- local implementation
- durable artifact creation
- skill updates
- continuity capture
- final repo truth before shared push

### Windows Claude/Cowork

Use for:

- planning
- architecture tradeoffs
- human-facing synthesis
- prompt shaping
- coordination across platforms and agents

### OpenClaw/Kimi

Use for:

- live runtime truth
- Linux services, cron, and gateway checks
- production-like validation
- live trading execution context

### Hermes

Use for:

- control-layer and adapter proof
- tighter-gated experiment lanes
- structured publish contracts and review gates

## Heartbeats And Automations

### Use A Heartbeat When

- the objective needs continuation across wakes
- approved work must resume after interruption
- periodic re-checks or stall detection are needed

### Do Not Use A Heartbeat When

- the work is a one-pass analysis or report
- the task is already bounded enough to finish in one pass
- the heartbeat would only duplicate a timer, hook, or manual handoff

### Rule Of Thumb

- timers and cron wake work
- hooks validate events
- heartbeats continue approved work or detect stalls
- orchestration decides, routes, and proves

## Visual Observability Recommendation

The best human-facing view is a small control-room pattern with:

- one top summary showing objective, class, phase, owner, and current gate
- one phase timeline from landscape through writing
- one worker fan-out view
- one conflict and rerun view
- one drill-down path from summary to evidence files

## Why The Verdict Schema Should Be Promoted

This run showed that the verdict schema changed cheap workers from vague recommenders into useful routing workers.

The high-value fields were:

- orchestration class
- why not other classes
- freshness requirement
- required artifacts
- required gates
- stop condition
- rerun policy
- escalation policy

## Final Decision

Promote the verdict schema deeper into:

- platform-specific OpenClaw workflow templates
- the Claude/Cowork adapter skill
- future worker prompt templates for swarm-backed work

Do not promote it by cloning giant platform-specific copies of the whole swarm skill.
Use small adapters and templates around the shared core instead.

## Remaining Limits

- this run proves the artifact contract and the policy flow more than cost savings
- it does not prove a live market swarm outcome
- it does not prove automatic dashboard rendering yet

Still, it is a real end-to-end proof that the swarm can start from a plan, move through dimensions and cross-verification, and end in a memo with clear policy rules.
