# Dimension 01 - Use-Case Routing

## Orchestration Class

- `deep research swarm`

## Why Not Other Classes

- not `direct task` because this is a policy and routing question, not one bounded implementation step
- not `bounded build` because the deliverable is a decision memo about where the swarm belongs
- not `always-on pipeline` because this slice is analyzing the live loop, not running it

## Freshness Requirement

- current-session truth for policy and shared skill files
- same-day or point-in-time refreshed truth for any live trading or runtime claims

## Required Artifacts

- `plan.md`
- `research/landscape_brief.md`
- `research/dim01.md` through `research/dimNN.md`
- `research/cross_verification.md`
- `research/targeted_validation.md` when needed
- `research/insights.md`
- `writing/outline.md`
- `final/report.md`

## Required Gates

- the task is genuinely research-heavy
- dimensions are distinct but still overlap
- required dimension outputs exist
- conflicts are classified explicitly
- uncertainty is preserved where evidence is weak
- the swarm is not treated as the default live trading loop

## Stop Condition

- stop when the routing boundary is explicit in the final memo
- stop on blocked if the run cannot produce a coherent use-case boundary
- stop after 3 no-progress wakes if the run ever becomes recurring

## Rerun Policy

- rerun only failed, weak, or ambiguous slices
- do not restart the whole swarm by default

## Escalation Policy

- raise reasoning first
- rerun only the disputed slice on a stronger lane when needed
- escalate to human review only if the work still cannot separate swarm-worthy work from routine work

## Core Findings

- deep swarm is justified when the question needs broad evidence gathering, contradiction handling, and a memo or thesis outcome
- it should be used for major trading theses, event-driven setup reviews, postmortems, large comparisons, and deep ambiguity that needs cross-verification
- it should not be used for routine trading cycles, position management loops, setup watchers, every five-minute or fifteen-minute wake, or normal coding tasks
- the live trading loop should stay lean; the swarm is a special research layer, not the default execution loop

## Explicit Do Use For

- weekly BTC or broader market theses
- major event-driven setup reviews
- deep ambiguity needing cross-verification
- post-trade failure analysis
- large token or sector comparisons
- thesis or memo work that needs contradiction handling

## Explicit Do Not Use For

- routine setup watcher work
- position management loops
- every five-minute or fifteen-minute market wake
- one-step monitor or repair tasks
- normal coding objectives
- routine trading cycles
