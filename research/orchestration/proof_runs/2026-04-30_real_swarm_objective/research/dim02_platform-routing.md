# Dimension 02 - Platform Routing

## Orchestration Class

- `deep research swarm`

## Why Not Other Classes

- not `direct task` because this is a cross-platform routing memo
- not `bounded build` because the output is a routing and proof contract, not a code patch
- not `always-on pipeline` because this is a one-time policy decision, not a recurring loop

## Freshness Requirement

- current shared skill files
- current workspace truth
- point-in-time verification for platform claims in this session

## Required Artifacts

- `plan.md`
- `research/landscape_brief.md`
- `research/dim02_platform-routing.md`
- `research/cross_verification.md`
- `research/insights.md`
- `final/chimera_deep_swarm_deployment_memo.md`

## Required Gates

- platform roles are distinct but share one core logic
- Codex, Claude/Cowork, OpenClaw/Kimi, and Hermes each have a concrete routing difference
- live trading loops stay lean
- proof output preserves contradictions and gaps

## Stop Condition

- stop on complete, blocked, or the explicit no-progress guard if recurrence ever gets added

## Rerun Policy

- rerun only ambiguous or conflicting platform slices
- do not restart the whole swarm unless the routing model itself is wrong

## Escalation Policy

- raise reasoning first
- escalate the model only for the failed slice
- use stronger review on contradictions or platform-discovery gaps

## Platform Recommendations

### Codex

- best for local implementation, durable file edits, skill migration, and continuity capture
- best place to turn routing decisions into shared artifacts and proof files

### Claude/Cowork

- best for planning, architecture tradeoffs, human-facing synthesis, and coordination
- should do pre-implementation shaping and policy framing

### OpenClaw/Kimi

- best for live runtime truth, service checks, cron and agent validation, and production-like execution proof
- should be the live operational lane, not the memo lane

### Hermes

- best for control-layer experiments and model or adapter proof
- should stay under explicit publish contracts and quality gates

## Shared Core Rules

- all platforms follow `plan -> execute -> review`
- the swarm is specialized, not the default for every task
- workers should be cheap-first; orchestrator, verifier, and reviewer should be stronger when judgment matters
- freshness should match claim type
- proof means artifacts, gates, and explicit stop conditions

## Platform-Specific Differences

- Codex emphasizes durable local artifact creation and cross-platform handoff files
- Claude/Cowork emphasizes orchestration design, synthesis, and decision framing
- OpenClaw/Kimi emphasizes live truth and runtime validation
- Hermes emphasizes control-layer proof and should use tighter publish gates
